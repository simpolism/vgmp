# Known bugs and issues

Findings from a full review of the Kotlin app layer and the JNI bridge (`vgmplayer_jni.cpp`),
2026-07-15. Ordered by severity within each section. Line numbers refer to the current `main`
(d4f6334); re-verify each against the file before fixing.

## Implementation status

- [x] Checkpoint 1: issues #1–#30 implemented and build-verified in the native/Kotlin correctness pass.

## Native / JNI (`app/src/main/cpp/vgmplayer_jni.cpp`)

### High

1. **Use-after-free of `filename` after `ReleaseStringUTFChars`** — lines ~600–612 (`nOpen` KSS
   branch), ~2942–2964 (`nGetKssTrackCountDirect`), ~3003–3024 (`nGetKssTrackRange`).
   `filename` points into the string returned by `GetStringUTFChars`; `ReleaseStringUTFChars` is
   called before `filename` is passed to `KSS_bin2kss(...)`. ART returns a heap copy, so this reads
   freed memory on every KSS open/scan (MBM detection is filename-based, so it can also misdetect).
   Fix: move the `Release` call after the last use (one-line reorder in each site).

2. **Use-after-free of `midiBuf` in `nGetTrackLengthDirect` MUS branch** — lines ~2128–2143.
   `mem_get_buf()` returns memio's internal buffer, `mem_fclose()` frees it
   (libMusDoom/src/memio.c frees `stream->buf` for write-mode files), then `adl_openData()` parses
   the freed buffer. The `nOpen` MUS branch does this correctly (copies into `gMusDoomMidiData`
   before closing, ~lines 770–774); mirror that here.

3. **PSF length scan races the background generation thread and leaks ~2.6 MB per call** —
   lines ~2188–2198. `sexy_load()` re-runs `psxInit()/psxReset()/SPUinit()` on *global* emulator
   state. The PSF generation thread (`gPsfGenerationThread`) is not covered by the Kotlin-side
   mutex, so scanning a PSF's length while a PSF is playing re-initializes the running
   interpreter's memory (data race, likely SIGSEGV). The path also never calls `psxShutdown()` on
   success, leaking `psxMemInit`'s ~2.6 MB per query. Fix: guard against scanning while generation
   is active (or add a metadata-only tag reader), and pair with `psxShutdown()`.

4. **GD3 tag parser out-of-bounds reads on malformed VGM files** — lines ~1611–1623 and
   ~1650–1661 (`readVgmGd3Tags`). (a) Only `gd3Ofs < eofOfs` is checked before reading the 4-byte
   magic at `gd3Ofs` and the 4-byte size at `gd3Ofs + 8`, so a GD3 offset within 12 bytes of EOF
   reads past the buffer. (b) With truncated/odd-sized GD3 data, `pos - start - 2` can underflow to
   ~4 GB and `utf16le_to_utf8` walks memory until it faults. A malformed file can crash the app.

5. **`NewStringUTF` called on raw tag bytes that are not Modified UTF-8** — lines ~1763 (libgme),
   ~1852–1926 (KSS, incl. `gKss->info[i].title`), ~1995 (PSF), ~1842 (libopenmpt).
   SPC/NSF/KSS tags are frequently Shift-JIS; PSF tags often Latin-1. Under CheckJNI this aborts;
   in release builds it's UB (corrupt strings/crashes). Add a sanitize/transcode helper in front of
   every `NewStringUTF` fed by file data. (libopenmpt returns standard UTF-8, which still differs
   from Modified UTF-8 for supplementary characters.)

### Medium

6. **PSF cache: ~202 MB up-front `reserve()`; invariant breaks for tracks over 20 minutes** —
   lines ~190–196 (`sexyd_update`), ~1310–1336 (`nFillBuffer`). `reserve(44100*4*1200)` can throw
   `std::bad_alloc` on the generation thread (no handler → `std::terminate`) on low-RAM devices.
   If generated audio exceeds the reserve (length tag > 20 min, or infinite-length files),
   `insert()` reallocates and invalidates the raw pointer `nFillBuffer` reads *after* dropping the
   lock → use-after-free race. The lock-free commit-counter scheme is only sound while the buffer
   never reallocates.

7. **`nSetChannelMuted` (libgme): unvalidated index** — lines ~2706–2713. Negative or
   out-of-range `index` reaches `gme_mute_voice` (libgme's bounds `require()` compiles out under
   NDEBUG → UB shift) and `gGmeMutedChannels[index]` (negative jint converts to huge `size_t` in
   the size comparison → OOB vector write). Early-return unless `0 <= index < gme_voice_count()`.

8. **Unchecked `ftell`/`fread` when sizing buffers** — lines ~585–591, ~662–667, ~741–747,
   ~2062–2067, ~2113–2118, ~2954–2959, ~3014–3019. `ftell` returning −1 yields
   `std::vector<uint8_t>(-1)` → `std::length_error` thrown through an `extern "C"` JNI frame →
   `std::terminate`. Two sites ignore the `fread` return entirely and process uninitialized bytes.

### Low

9. Header debug logs read 8 bytes without a size check — lines ~606–608 (KSS), ~756–758 (MUS);
   files shorter than 8 bytes cause OOB reads before format validation.
10. Missing null checks on `Get*ArrayElements` — lines ~1210 (`nFillBuffer`), ~1527
    (`nGetChannelSpectrums`); OOM would SIGSEGV instead of failing cleanly.
11. `nGetSpectrum` writes 512 floats without checking the array length — lines ~1459–1472.
    Currently safe (Kotlin always passes 512) but one `GetArrayLength` guard away from robust.
12. KSS seek: negative `samplePos` becomes ~2^64 in `UINT64 remaining` → effectively infinite
    fast-forward while holding the engine mutex (audio + UI freeze) — lines ~1186–1192. Valid
    seeks also synchronously emulate the full target duration under the mutex. Dead `gVgmPlayer`
    check at ~1171–1174.
13. PSF `nPlay` resets position to 0 — lines ~862–865 — so play-after-pause restarts PSF tracks
    from the beginning while every other backend resumes.
14. PSF underrun check off-by-one (`committed <= currentPos + 4` should be `<`) — line ~1325.
    Cosmetic, end of track only.

## Kotlin app layer

### High

15. **Seeking backwards during the fade-out window still ends the track** —
    `VgmPlaybackService.kt:716` (`seekTo`) never resets `isFadingOut`/`fadeStartTimeMs`. Once the
    fade has started (last 2 s of a track, `VgmPlaybackService.kt:594`), a user seek to an earlier
    position keeps the fade running: audio stays faded to silence and the end-detection at
    `VgmPlaybackService.kt:599` (`isFadingOut && now >= fadeStartTimeMs + FADE_MS`) skips to the
    next track ~2 s later regardless of the new position. Fix: in `seekTo`, reset `isFadingOut`
    (and consider re-deriving it from the new position).

16. **Stale chip-volume and channel UI when consecutive tracks have the same count** —
    `NowPlayingFragment.kt:515` and `NowPlayingFragment.kt:559` rebuild the sound-chip sliders and
    channel mute/solo rows only when `container.childCount != count`. Two different tracks with
    the same number of devices/channels (very common: any two 2-chip VGMs, any two 4-channel MODs)
    keep the previous track's chip names, slider positions, and channel names. Fix: also compare a
    signature of the names (or the track key, which is already computed at
    `NowPlayingFragment.kt:544`) before skipping the rebuild.

### Medium

17. **Position tracking is wall-clock based and drifts from audio** —
    `VgmPlaybackService.kt:727` computes position as `elapsedRealtime - playbackStartTimeMs`.
    Whenever `fillBuffer` returns 0 (PSF generation stalls, `VgmPlaybackService.kt:568`) or the
    AudioTrack underruns, the wall clock runs ahead of the audible audio; the deep device buffer
    (`minBuf * 4`, `VgmPlaybackService.kt:326`) adds a constant lead. Consequences: seekbar ahead
    of audio, and the wall-clock-triggered fade at `VgmPlaybackService.kt:594` can start early and
    cut real audio. Fix: derive position from `VgmEngine.getCurrentSample()` (already exposed) or
    `AudioTrack.getPlaybackHeadPosition()`.

18. **ZIP cache is unbounded and never invalidated** — `ZipArchiveStore.kt:27–43`. Every browsed
    ZIP is copied whole into `cacheDir/zip-archives`, keyed only by a hash of the URI. If the
    user replaces/updates the ZIP, the stale cached copy is used forever; and nothing evicts old
    archives, so the cache grows without bound. Fix: include size + last-modified from the
    content resolver in the cache key (invalidation) and add an LRU/size cap or clean-on-start.

19. **`onDestroy` can cancel the engine shutdown it just requested** —
    `VgmPlaybackService.kt:874–879`: `stopPlayback()` launches `VgmEngine.stop()/close()` on
    `serviceScope`, and the next line calls `serviceScope.cancel()`, which can cancel that
    coroutine before it runs — the native engine (and the PSF generation thread) may never be
    closed on service destroy. Fix: close the engine synchronously (e.g. `runBlocking` with a
    timeout) or use a scope that isn't cancelled until after cleanup.

20. **Duplicate flow collectors on the player screen** — `NowPlayingFragment.observePlaybackInfo`
    is called from both `onViewCreated` (`NowPlayingFragment.kt:62`) and `onServiceConnected`
    (`NowPlayingFragment.kt:80`), and nothing cancels the first set, so the fragment can end up
    with two collectors per flow (doubled `updateUI` work, and it makes bug #21 easier to hit).
    `MainActivity.kt:58` similarly launches a new `playbackInfo` collector on every service
    (re)connection without cancelling the previous one.

21. **Channel VU-meter rebuild race** — `NowPlayingFragment.kt:120–143`: when the channel count
    changes, `removeAllViews()` runs synchronously but the rows are added inside a launched
    coroutine. Spectrum frames arrive at 30 Hz, so a second frame can observe `childCount == 0`,
    launch a second builder, and both append rows → duplicated meters until the next mismatch.
    Fix: build synchronously, or guard with an "is rebuilding" flag / track-key check.

22. **Media button events are dropped when the service isn't running** — the manifest declares
    `androidx.media.session.MediaButtonReceiver`, which will start `VgmPlaybackService` (fallback
    to the MediaBrowserService), but `onStartCommand` (`VgmPlaybackService.kt:863`) only handles
    the custom `ACTION_*` strings and never calls `MediaButtonReceiver.handleIntent(mediaSession,
    intent)`, so a Bluetooth play press with the app swiped away starts the service and then does
    nothing. Fix: call `MediaButtonReceiver.handleIntent` in `onStartCommand`.

### Low

23. **`playQueue` start-index mismatch after filtering** — `VgmPlaybackService.kt:344–349` filters
    unsupported tracks but applies the caller's `startIndex` to the filtered list. Current callers
    pass pre-filtered lists so it's latent, but any playlist entry that stops being supported
    would shift which track plays. Also, when *everything* is filtered out it's a silent no-op.
24. **Shuffle can repeat the current track** — `PlaybackQueue.kt:31` `moveRandom()` draws from all
    indices including the current one; with small queues "next" in shuffle frequently replays the
    same track. No shuffle history for "previous" either.
25. **Fade math nits** — `VgmPlaybackService.kt:620` clamps the fade factor to `0f..100f` (upper
    bound should be `1f`; currently unreachable but wrong), and `currentVolume`
    (`VgmPlaybackService.kt:92`) is dead.
26. **Idle foreground notification** — the service calls `startForeground` in `onCreate`
    (`VgmPlaybackService.kt:142`), so a "VGMP" media notification appears as soon as the app is
    opened, before anything plays, and stays until playback stops. Consider demoting to
    foreground only when playback starts (`startForeground` from `startTrackWithFocus`) and
    `stopForeground(STOP_FOREGROUND_DETACH)` on pause.
27. **Android Auto browse icons ignore real artwork** — `VgmPlaybackService.kt:846`
    (`getScaledArtForAuto`) always scales the fallback logo, allocating a new bitmap per item in
    the `onLoadChildren` loop; playlist/track rows never show actual art.
28. **`updatePlaybackState(STATE_STOPPED)` reports a garbage position** — `currentPositionMs()`
    keeps computing from the stale `playbackStartTimeMs` after stop (`VgmPlaybackService.kt:727`).
29. **`previousTrack` cuts abruptly while `nextTrack` fades** — `VgmPlaybackService.kt:696–715`;
    inconsistent, and previous also doesn't do the usual "restart current track if > 3 s in".
30. **Seek bar has 100 steps** — `NowPlayingFragment.kt:252` maps progress 0–100 to the duration,
    so on a long track a single step is several seconds. Use ms-resolution max instead.
